Aperiodic Database Query

     

In order for an information server effectively to query the large number of heterogeneous passive information sources in the Matrix, it requires a model which accounts for the following factors:        

  • the rate that new information becomes available for query at a         remote database
  • the delay between issuing a query and obtaining a result
  • the probability that a query message or its result message will be lost    

In the following query scheduling  model , we assume that information somehow flows  into passive  WAN database systems, and is then made available for query. An optimal query schedule will allow the extractor  program, that is the information service which is querying the remote WAN database, to obtain information at the same rate as it arrives at the server. Obviously some delay  is inevitable since the source information is not instantly available. We assume that the rate of information flow however, is approximately the same over some time interval. Querying too often would result in many empty or redundant  answers. Querying too infrequently would result in large, infrequent ``bursts '' of information  at the extractor, and large intervals of time where the extractor's information  is out of date. More importantly, part of the information in such bursts is already old when they arrive at the extracting system. Since WAN communications bandwidth  can be expensive, and since the clients of every SDI service prefer current information, neither of these situations is desirable.

Measuring the arrival rate, i.e. the rate at which information arrives at the passive server is the usual approach. However, from the extractor's point of view, a more important parameter is the rate that the information becomes available for querying. These two rates are not identical because the passive database source server must update  the information to make it available for query. We shall call this update rate   and define it to be the number of  information items which the server makes available for querying per unit time. This rate may indeed be different from the arrival rate, but since the extractor has no control over the indexing process at the passive database source, this latter parameter must take precedence. Thus, using the extractor's perspective we shall take a phenomenological ``black box'' approach. 

Without loss of generality, we consider each passive WAN server (figure 2.1) separately, and optimize a scheduling rate for an individual one. Let   (with units of queries per unit time) be the rate at which the extractor sends queries to the server. The parameter is designed to allow an extractor to mirror a remote information   source. Hence, it is desirable that an extractor    querying at a rate retrieve information as soon as it becomes available. Therefore, if is optimal it will allow an extractor to retrieve, on average, precisely one message per query. Retrieving more than one message decreases the timeliness  and therefore the isomorphism of the model . If messages queue  at a server and become available for query in groups, it is impossible for a single query to return exactly one information item; it is, however, possible for the average query rate  over time to produce one item per query. Bursts of available items at a server will lead to cases where this definition of an optimal would cause a system    to send several queries which retrieve zero items. Defining an optimal to be a rate which allows an extractor to retrieve items just after each update  at a remote server would therefore be more efficient. However, experience indicates that burst sizes are usually very small, and that sporadic updates rates, large burst sizes, or server unavailability are not common; therefore optimizing for one item retrieved per query provides a more useful method than one query per server update.

If, on average, an    extractor querying at a rate retrieves more than one information item, the rate is too slow. If no new information items are retrieved, the rate is too fast. Thus far, our analysis assumes that no errors or network failures     can occur, an assumption that is, of course, completely unrealistic. Network failures cannot be ignored; we are therefore obliged to add recovery factors and robustness to the model for the unfortunate case of network breakdowns.
